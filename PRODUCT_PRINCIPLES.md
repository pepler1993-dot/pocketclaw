# Product Principles

## 1. Simple by default
The default experience should be understandable to non-technical users.

## 2. Advanced when needed
Power users should still be able to access deeper controls, but those controls should not dominate the interface.

## 3. Local first
Whenever possible, the product should favor local execution, local storage, and local control.

## 4. Privacy matters
Users should understand what runs locally, what talks to cloud services, and what data leaves the device.

## 5. Mobile-first experience
This is not a desktop control panel squeezed onto a phone. It should feel native to mobile usage patterns.

## 6. Fast setup
Users should be able to reach a usable assistant quickly through onboarding, templates, and sane defaults.

## 7. Voice is core, not a gimmick
Speech input and spoken output should feel like part of the product, not an afterthought.

## 8. Clear language over technical jargon
Prefer terms users understand over internal OpenClaw terms wherever possible.

Examples:
- "Only this device" instead of "loopback"
- "Available on local network" instead of "LAN bind"
- "Connect Telegram" instead of low-level channel jargon

## 9. Reliability over feature spam
A small set of features that work well is better than an impressive but fragile feature list.

## 10. Android first, iPhone carefully
Android is the most realistic path for a true local-first runtime. iPhone support should be designed honestly around platform limits.
